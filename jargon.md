## 1. SDK 핵심 인터페이스

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **Claude Agent SDK** | Claude Code의 에이전트 루프·도구 실행·세션 관리를 라이브러리로 제공하는 SDK. Python(`claude-agent-sdk`)과 TypeScript(`@anthropic-ai/claude-agent-sdk`) 두 버전이 있다. | 직접 API를 호출하고 도구 루프를 구현해야 한다. 도구 실행 코드만 수백 줄이 필요하다. | 반복적인 도구 루프 구현 없이 `query()` 한 줄로 자율 에이전트를 실행할 수 있다. | Anthropic Client SDK + 직접 도구 루프 구현 | 자동차 vs 바퀴·엔진 부품 조립. SDK는 완성차, Client SDK는 부품 세트다. |
| **`query()`** | 에이전트에게 프롬프트를 전달하고 응답 메시지들을 비동기 제너레이터로 반환하는 핵심 함수. 호출 한 번이 에이전트 루프 한 사이클이다. | 에이전트를 실행할 방법이 없다. | "prompt 전달 → 도구 호출 반복 → 최종 응답"까지의 전체 루프를 단일 인터페이스로 추상화한다. | `ClaudeSDKClient.query()` (멀티-턴용), Anthropic Client SDK의 `messages.create()` | 식당에서 주문하기. `query()`는 웨이터에게 주문서를 건네는 행위. 요리(도구 실행)는 주방이 알아서 한다. |
| **`ClaudeSDKClient`** | 여러 번의 대화를 하나의 세션으로 유지하는 클라이언트 객체. `async with` 컨텍스트 매니저로 사용하며, `query()`와 달리 호출 사이에 세션이 유지된다. | 멀티-턴 대화 시 매번 `session_id`를 직접 추적해 `resume`에 넘겨야 한다. | "이전 대화 기억"이 필요한 챗봇·인터랙티브 에이전트에서 세션 관리를 자동화한다. | `query()` + `resume=session_id` 수동 관리 | 전담 비서. `query()`가 매번 다른 직원에게 업무를 맡기는 것이라면, `ClaudeSDKClient`는 같은 담당자에게 지속적으로 업무를 위임하는 것이다. |
| **`ClaudeAgentOptions`** | `query()`나 `ClaudeSDKClient`에 전달하는 설정 객체. 허용 도구, 시스템 프롬프트, 권한 모드, MCP 서버 등 모든 에이전트 동작을 제어한다. | 모든 도구가 허용되고 권한 확인 없이 실행된다. 원하지 않는 파일이 수정될 수 있다. | 에이전트 동작 범위와 권한을 선언적으로 정의해 예측 가능한 실행을 보장한다. | 없음 (이 객체 자체가 설정 인터페이스) | 직원에게 업무 지시할 때 쓰는 업무 범위 명세서. "이것만 해도 되고, 저건 하지 마라"를 문서화한 것이다. |
| **`ANTHROPIC_API_KEY`** | Anthropic API를 인증하는 비밀 키. 환경 변수로 설정하며 SDK가 자동으로 읽는다. | API 호출이 401 인증 오류로 실패한다. | 누가 API를 호출하는지 식별해 요금 청구와 사용량 제한을 가능하게 한다. | Bedrock(AWS 자격증명), Vertex AI(GCP 자격증명), Azure Foundry | 건물 출입증. 없으면 문이 열리지 않는다. |

## 2. 메시지 타입

`query()`가 yield하는 메시지들의 종류. 에이전트 루프 진행 상황을 나타낸다.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **`AssistantMessage`** | Claude가 텍스트 응답이나 도구 호출을 생성할 때마다 yield되는 메시지. `content` 배열에 블록들이 담긴다. 도구 사용 중에는 여러 번 올 수 있다. | Claude 응답을 화면에 표시할 수 없다. | 에이전트 루프의 각 응답 단계를 개별적으로 처리할 수 있게 한다. | `StreamEvent` (더 세밀한 실시간 처리가 필요할 때) | 회의 중 발언자의 말 한 마디. 여러 차례 발언할 수 있으며, 최종 결론이 나오기 전에도 중간 발언을 들을 수 있다. |
| **`ResultMessage`** | 에이전트 루프가 완전히 끝났을 때 마지막으로 오는 메시지. 성공/실패 여부, 총 비용, 세션 ID, 최종 결과 텍스트를 포함한다. | 작업이 완료됐는지 알 수 없고 세션 ID도 얻지 못한다. | 에이전트 루프의 종료 시점과 결과를 명확히 알려준다. 비용 추적·세션 재개에도 필수다. | 없음 (루프 종료 신호로 대체 불가) | 택배 배송 완료 문자. 물건(응답)이 오는 동안은 상태 메시지가 오고, 배달이 끝나면 완료 알림이 온다. |
| **`SystemMessage`** | SDK 내부 이벤트를 전달하는 메시지. `subtype="init"`이면 세션 시작 신호이며 세션 ID와 연결된 MCP 서버 목록이 담겨 있다. | MCP 서버 연결 성공 여부를 확인할 수 없다. | 세션 시작·초기화 상태를 프로그래밍적으로 감지할 수 있게 한다. | 없음 | 회의 시작 전 진행자가 "오늘 참석자와 안건을 확인하겠습니다"라고 하는 도입 멘트. |
| **`UserMessage`** | 사용자 입력이 에코(echo)될 때 오는 메시지. 일반적으로 직접 처리할 필요는 없다. | 없어도 에이전트 실행에 지장 없음 | 대화 기록 재구성이나 디버깅 시 입력을 추적할 수 있다. | 없음 | 전화 통화 녹음에서 상대방 말 전에 내 말도 함께 녹음되는 것. |
| **`StreamEvent`** | `include_partial_messages=True` 설정 시만 yield된다. Claude가 텍스트를 생성하는 **중간 중간**에 글자 단위로 도착한다. `event` 필드에 원시 API 스트림 이벤트가 담긴다. | 텍스트가 완성될 때까지 화면이 비어 있다. | 응답이 생성되는 즉시 화면에 표시해 사용자 체감 속도를 높인다. | 폴링(주기적으로 결과 확인) — 구현이 복잡하고 지연 발생 | 라이브 자막. 발화가 끝나기 전에 말하는 내용이 실시간으로 화면에 표시되는 것. |
| **`RateLimitEvent`** | API 사용량이 제한에 가까워지거나 초과했을 때 오는 메시지. `status` 필드로 허용/경고/차단 상태를 알 수 있다. | 속도 제한에 걸려도 이유를 모른다. 그냥 실패로 보인다. | 속도 제한 상태를 감지해 재시도·대기 로직을 구현할 수 있다. | 없음 | 도로 속도 제한 표지판. 표지판 없이 단속 카메라만 있으면 왜 걸렸는지 모른다. |

## 3. Content 블록

`AssistantMessage.content` 배열 안에 들어 있는 개별 블록 타입들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **`TextBlock`** | Claude가 생성한 일반 텍스트 응답. `text` 필드에 문자열이 담긴다. 가장 자주 만나는 블록이다. | 응답 텍스트를 출력할 수 없다. | 텍스트 응답과 도구 호출을 같은 `content` 배열에서 타입으로 구분해 처리한다. | 없음 | 편지지에 쓴 본문. 봉투(AssistantMessage) 안에 들어 있는 실제 편지 내용. |
| **`ToolUseBlock`** | Claude가 도구를 호출할 때 생성되는 블록. `name`(도구명), `input`(인자), `id`(추적용 ID)를 포함한다. | 어떤 도구가 어떤 인자로 호출됐는지 알 수 없다. | 도구 호출 내역을 로깅하거나 특정 도구 사용을 감지·차단하는 데 사용한다. | `PreToolUse` 훅 (더 강력한 제어가 필요할 때) | 직원이 업무 처리를 위해 특정 시스템에 로그인했다는 기록. 어떤 시스템을, 어떤 목적으로 접근했는지 남긴다. |
| **`ToolResultBlock`** | 도구 실행 결과가 다시 Claude에게 전달될 때의 블록. `tool_use_id`로 어떤 도구 호출의 결과인지 연결된다. | 도구 실행 결과를 추적하거나 오류 여부를 확인할 수 없다. | 도구 실행 성공/실패를 감지하고 결과를 파이프라인 로직에 활용한다. | 없음 | 심부름을 보낸 뒤 돌아온 결과 보고서. "마트에 가서 우유 사왔습니다(또는 품절이었습니다)." |
| **`ThinkingBlock`** | Extended Thinking 기능을 활성화했을 때만 나타나는 블록. Claude의 내부 추론 과정이 텍스트로 담긴다. | Extended Thinking 비활성화 시 아예 나타나지 않음 | 복잡한 추론이 필요한 작업에서 Claude가 더 깊이 생각하게 하고, 그 과정을 관찰할 수 있게 한다. | `system_prompt`에 단계별 사고 지시 | 수학 시험지의 풀이 과정. 정답(TextBlock)이 맞더라도 어떻게 생각했는지 보여준다. |

## 4. 빌트인 도구

`allowed_tools`에 이름을 추가하는 것만으로 Claude가 사용할 수 있는 내장 도구들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **`Read`** | 파일 경로를 받아 내용을 읽는 도구. 코드 파일, 설정 파일, 문서 등 어떤 텍스트 파일이든 읽을 수 있다. | Claude가 파일 내용을 알 수 없어 코드베이스를 이해하거나 수정할 수 없다. | 로컬 파일 시스템을 Claude의 컨텍스트로 가져온다. | MCP filesystem 서버, 파일 내용을 직접 프롬프트에 붙여넣기 | 도서관에서 책을 꺼내 읽는 것. 서재(파일 시스템)에서 필요한 책(파일)을 가져온다. |
| **`Write`** | 지정된 경로에 새 파일을 생성한다. 이미 존재하는 파일은 덮어쓴다. | Claude가 새 파일을 만들 수 없다. | 새 코드 파일, 보고서, 설정 파일 생성을 자동화한다. | `Edit`(기존 파일 수정), `Bash`의 리다이렉션(`echo > file`) | 빈 종이에 새 문서를 작성하는 것. |
| **`Edit`** | 기존 파일의 특정 문자열을 찾아 다른 문자열로 교체한다. 파일 전체를 다시 쓰지 않고 정확한 위치만 수정한다. | 파일 수정 시 전체를 다시 써야 해 실수가 생기고 맥락을 잃기 쉽다. | 대형 파일에서 정확한 최소 변경을 가능하게 한다. 변경 추적이 명확하다. | `Write`(전체 재작성), `Bash`의 `sed` | 문서에서 특정 단어를 찾아 바꾸기(Ctrl+H). 전체를 다시 쓰지 않고 필요한 부분만 수정한다. |
| **`Bash`** | 쉘 명령어를 실행한다. git, npm/bun, pytest, curl 등 터미널에서 실행 가능한 모든 명령을 Claude가 직접 실행할 수 있다. | 코드를 작성해도 실행하거나 테스트할 수 없다. CI/CD 작업 불가. | 코드 작성 → 테스트 실행 → 결과 확인을 하나의 에이전트 루프 안에서 완결할 수 있다. | 개별 도구(Read/Write)의 조합, MCP를 통한 특화 실행 환경 | 만능 칼. 조리, 자르기, 여는 것 등 다양하게 쓸 수 있지만 그만큼 주의가 필요하다. |
| **`Glob`** | 와일드카드 패턴으로 파일을 찾는 도구. `**/*.ts`, `src/**/*.py` 같은 패턴을 지원한다. | Claude가 어떤 파일이 있는지 파악하려면 `Bash`의 `find`를 써야 한다. | 특정 확장자나 경로 패턴의 파일 목록을 빠르게 얻는다. | `Bash`의 `fd` / `find` 명령어 | 서랍 안에서 "파란색 봉투"만 전부 찾아 꺼내오라는 지시. |
| **`Grep`** | 파일 내용을 정규식으로 검색한다. 특정 함수명, 패턴, 문자열이 어떤 파일 어느 줄에 있는지 찾는다. | 코드베이스에서 특정 패턴을 찾으려면 `Bash`의 `rg`나 `grep`을 써야 한다. | 대형 코드베이스에서 관련 코드 위치를 정확히 파악한다. 리팩토링 범위 파악에 필수. | `Bash`의 `rg`(ripgrep) / `grep` 명령어 | 책에서 특정 단어가 몇 페이지에 나오는지 색인으로 찾는 것. |
| **`WebSearch`** | 웹을 검색해 최신 정보를 가져온다. Claude의 학습 컷오프 이후 정보나 실시간 데이터를 조회할 수 있다. | Claude는 학습 데이터 기준일 이후의 정보를 모른다. 최신 라이브러리 버전, 최근 이슈 파악 불가. | 시간에 민감한 정보나 실시간 데이터가 필요한 작업을 처리한다. | `WebFetch`(특정 URL 직접 접근), MCP의 특화 서버 | 도서관 사서에게 "최근 논문 찾아달라"고 하는 것. 직접 검색하지 않아도 된다. |
| **`WebFetch`** | 특정 URL의 웹 페이지를 가져와 파싱한다. HTML을 마크다운으로 변환해 Claude가 읽기 쉬운 형태로 제공한다. | 특정 문서 URL의 내용을 Claude에게 읽힐 수 없다. | 공식 문서, API 레퍼런스, 특정 웹 페이지를 Claude 컨텍스트로 가져온다. | `WebSearch` + URL 선택, MCP playwright 서버(동적 페이지) | 특정 신문 기사 링크를 직접 열어 읽는 것. |
| **`AskUserQuestion`** | 에이전트 실행 도중 사용자에게 선택지를 제시하고 답변을 기다린다. | 에이전트가 애매한 상황에서 임의로 결정하거나 중단한다. | 중요한 결정 시점에 사람이 개입할 수 있게 해 Human-in-the-Loop 워크플로우를 구현한다. | 프롬프트 엔지니어링으로 Claude 스스로 결정하게 함 | 의사가 수술 전 "A안과 B안 중 어느 방법을 선택하시겠습니까?"라고 환자에게 묻는 것. |
| **`Monitor`** | 백그라운드에서 실행 중인 프로세스의 출력을 실시간으로 감시한다. 서버 시작, 빌드 완료 등의 이벤트에 반응한다. | 긴 빌드/서버 시작을 기다리려면 `Bash`에서 sleep을 쓰거나 폴링해야 한다. | "서버가 준비되면 테스트를 시작해라" 같은 이벤트 기반 로직을 구현한다. | `Bash` + sleep/폴링 루프, 외부 큐(Redis pub/sub) | CCTV 모니터링 요원. 특정 이벤트(침입 감지)가 발생할 때까지 화면을 지켜보다 즉각 반응한다. |

## 5. 세션 & 히스토리 관리

대화 맥락이 어떻게 보존되고 재사용되는지에 관한 개념들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **Session (세션)** | 에이전트 루프 한 사이클 이상에 걸쳐 누적된 **대화 기록**. 프롬프트, 도구 호출, 도구 결과, 응답 전부를 포함하며 `~/.claude/projects/<cwd>/<id>.jsonl` 파일에 자동 저장된다. | 매번 새 대화로 시작해 앞에서 읽은 파일·분석 내용을 Claude가 기억하지 못한다. | 긴 작업을 여러 단계로 나눠도 이전 단계의 지식이 유지된다. 프로세스 재시작 후에도 이어받을 수 있다. | 프롬프트에 이전 결과를 직접 삽입(수동 컨텍스트 관리), Redis에 대화 기록 저장 | 노트가 보존되는 회의실. 오늘 회의 내용이 다음 회의 때도 화이트보드에 남아 있다. |
| **`session_id`** | 세션을 고유하게 식별하는 UUID 문자열. `ResultMessage`와 `SystemMessage(init)`에서 얻을 수 있다. | 특정 세션을 지목해서 재개하거나 관리할 수 없다. | 특정 세션으로 정확히 되돌아가거나 외부 DB(Redis 등)에 저장해 사용자별 세션을 관리할 수 있다. | 없음 (SDK 내부 식별자) | 은행 업무 접수 번호. 이 번호로 내 업무 진행 상황을 조회하고 이어서 처리할 수 있다. |
| **`resume`** | 특정 `session_id`를 지정해 해당 세션을 이어받는 옵션. 프로세스가 재시작되어도 동일 session_id이면 이전 맥락이 복원된다. | 프로세스를 재시작하면 처음부터 다시 파일을 읽고 분석해야 한다. | "지난번에 분석한 내용에서 이어서 작업해라"가 가능해진다. | `continue_conversation=True`(가장 최근 세션 자동 이어받기) | 책갈피. 어제 읽던 페이지에 꽂아둔 책갈피 덕분에 처음부터 다시 읽지 않아도 된다. |
| **`continue` / `continue_conversation`** | `resume`과 달리 session_id를 몰라도 **가장 최근 세션**을 자동으로 이어받는 옵션. 단일 프로세스에서 여러 `query()` 호출 간 컨텍스트를 유지할 때 편리하다. | `session_id`를 매번 저장하고 전달해야 한다. | 단일 프로세스 내 멀티-턴 대화에서 session_id 관리 코드를 제거한다. | `ClaudeSDKClient`(더 명시적인 방식), `resume=session_id`(수동) | 열어둔 탭. 브라우저를 닫지 않는 한 이전에 보던 페이지 상태가 그대로다. |
| **`fork_session`** | 기존 세션을 **복사**해 새 세션을 만든다. 원본 세션은 그대로 유지되고, 새 세션은 독립적으로 다른 방향을 탐색한다. | 다른 접근 방법을 시도하면 원본 맥락을 잃는다. | "JWT로 구현"과 "OAuth2로 구현"을 동시에 시도하고 나중에 비교할 수 있다. | 별도의 `query()` 호출에 프롬프트 복제(불완전한 대체) | Git 브랜치. `main`에서 `feature/jwt`와 `feature/oauth2`로 나뉘어 각자 개발하는 것. |
| **`list_sessions()`** | 현재 디렉토리에서 생성된 세션 목록을 반환한다. 세션 ID, 요약, 생성 시간, 마지막 수정 시간 등을 제공한다. | 어떤 세션들이 있는지 코드로 알 수 없어 세션 선택기 UI를 만들 수 없다. | 세션 목록을 프로그래밍적으로 관리하고 사용자에게 선택지를 제공할 수 있다. | 파일 시스템에서 `~/.claude/projects/.../*.jsonl` 직접 탐색 | 클라우드 저장소의 "최근 문서" 목록. |
| **`rename_session()`** | session_id로 세션의 표시 제목을 변경한다. | 모든 세션이 자동 생성된 UUID나 짧은 요약으로만 식별된다. | "서울 인구 분석 세션"처럼 의미 있는 이름을 붙여 관리한다. | 없음 | 파일 이름 바꾸기. `IMG_20240101.jpg`를 `가족여행_제주도.jpg`로 바꾸는 것. |
| **`tag_session()`** | 세션에 태그 문자열을 붙인다. 나중에 태그로 세션을 필터링할 수 있다. | 세션이 쌓이면 목적별로 구분하기 어렵다. | "geography", "code-review" 같은 태그로 세션을 분류해 검색과 관리를 쉽게 한다. | `rename_session()`으로 태그를 제목에 포함시키기 | 파일에 색상 태그 붙이기. 프로젝트별, 주제별로 색깔을 달리해 빠르게 구분한다. |

## 6. 권한 & 보안

에이전트가 무엇을 할 수 있고, 할 수 없는지를 제어하는 메커니즘.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **`allowed_tools`** | 사용자 확인 없이 자동 실행이 허용된 도구 목록. 여기에 없는 도구를 Claude가 쓰려 하면 승인 프롬프트가 뜨거나 차단된다. | 에이전트가 Bash로 `rm -rf` 같은 위험한 명령을 실행할 수 있다. | **최소 권한 원칙(Least Privilege)**: 에이전트가 작업에 꼭 필요한 도구만 쓸 수 있게 제한한다. | `permission_mode="bypassPermissions"` (위험, 모두 허용) | 신입 직원 권한 목록. "읽기는 되고, 삭제는 안 된다"는 접근 제어 목록(ACL). |
| **`disallowed_tools`** | `allowed_tools`와 반대로, 명시적으로 **차단**할 도구 목록. | 에이전트가 원치 않는 도구를 사용할 수 있다. | `allowedTools`에 모든 도구를 나열하는 대신 특정 도구만 차단할 때 편리하다. | `allowed_tools`에 허용 목록만 정의 (화이트리스트 방식) | 블랙리스트. "이것만 빼고 다 해도 된다." |
| **`permission_mode`** | 도구 실행 시 사용자 확인 방식을 결정하는 모드. `"default"`, `"acceptEdits"`, `"plan"`, `"dontAsk"`, `"bypassPermissions"` 다섯 가지가 있다. | 모든 도구 실행마다 대화형 승인 프롬프트가 뜬다. CI/CD 환경에서는 차단된다. | 운영 환경(자동 승인)과 개발 환경(매번 확인)을 동일 코드로 전환할 수 있다. | `allowed_tools`로 특정 도구만 자동 승인 | 건물 보안 수준 설정. 방문객 모드(모두 확인), 직원 모드(카드만 있으면 통과), VIP 모드(아무 제한 없음). |
| **`acceptEdits`** | `permission_mode`의 한 값. 파일 편집(`Edit`, `Write`) 도구는 자동 승인하되, 나머지(`Bash` 등)는 여전히 확인한다. | 파일 수정마다 매번 승인 클릭이 필요하다. | "파일은 수정해도 되지만 명령어 실행은 확인받아라" 수준의 세밀한 제어가 가능하다. | `allowed_tools=["Edit", "Write"]`로 특정 도구만 허용 | 계약서 서명 권한. 특정 금액 이하는 팀장이 바로 서명하고, 그 이상은 임원 결재를 받는 것. |
| **`bypassPermissions`** | 모든 권한 확인을 생략한다. CI/CD 파이프라인이나 완전 자동화 환경에서 사용한다. | CI 환경에서 대화형 프롬프트를 기다리다 타임아웃이 발생한다. | 사람이 없는 자동화 환경에서 에이전트가 막히지 않고 실행된다. | `allowed_tools`에 필요한 도구 모두 나열 (더 안전한 방법) | 24시간 무인 공장. 사람의 승인 없이 모든 기계가 자동으로 돌아간다. 보안은 사전 설계로 보장해야 한다. |
| **`can_use_tool` 콜백** | 도구 실행 전에 호출되는 커스텀 함수. 도구명과 입력값을 받아 허용/거부를 동적으로 결정할 수 있다. | 정적인 `allowed_tools` 목록으로는 "파일 경로가 `/system/`이면 차단" 같은 동적 규칙을 만들 수 없다. | 런타임 조건(파일 경로, 명령어 내용 등)에 따라 도구 실행 여부를 동적으로 결정한다. | `PreToolUse` 훅 (유사하지만 차단 방식이 다름) | 경비원. 출입증만 확인하는 게 아니라 "오늘 예약이 있는지", "방문 목적이 뭔지"도 물어보는 동적 판단. |

## 7. MCP — Model Context Protocol

에이전트가 외부 시스템과 통신하기 위한 표준 프로토콜.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **MCP** | Model Context Protocol. AI 에이전트와 외부 도구·데이터 소스를 연결하는 **오픈 표준 프로토콜**. Anthropic이 설계했으며 GitHub, Slack, DB 등 수백 가지 서버가 있다. | 외부 서비스 연동마다 커스텀 도구를 직접 구현해야 한다. | "한 번 만들어 어디서든 쓰는" 표준 인터페이스로 도구 재사용성을 극대화한다. | REST API 직접 호출 + 커스텀 도구 구현 | USB 표준. USB가 없던 시절엔 프린터마다, 마우스마다 전용 포트가 달랐다. MCP는 AI 에이전트용 USB다. |
| **MCP 서버** | MCP 프로토콜을 구현한 프로그램. 특정 서비스(GitHub, PostgreSQL 등)와의 통신 로직을 담당하며, 에이전트에게 도구 목록을 제공한다. | 해당 서비스의 API를 직접 호출하는 코드를 작성해야 한다. | 서비스별 API 차이를 추상화해 Claude가 동일한 방식으로 다양한 서비스를 사용할 수 있게 한다. | 직접 구현한 SDK 내장 도구(`@tool` 데코레이터) | 통역사. Claude는 영어만 하고, 각 서비스는 자체 언어를 쓴다. MCP 서버가 통역을 담당한다. |
| **stdio transport** | MCP 서버를 **로컬 프로세스**로 실행하고 표준 입출력(stdin/stdout)으로 통신하는 방식. `command`와 `args`로 설정한다. | 로컬 파일 시스템이나 설치된 프로그램을 MCP로 연결할 방법이 없다. | npx, uvx 등으로 즉시 실행 가능한 로컬 MCP 서버들을 간단하게 연결한다. | HTTP/SSE transport(원격), 직접 구현한 SDK 도구 | 같은 회사 내부 직원에게 전화 대신 옆자리에서 직접 말하는 것. 네트워크 없이 빠르고 안정적이다. |
| **HTTP/SSE transport** | MCP 서버가 **원격 URL**에서 실행될 때 사용하는 방식. HTTP(단순 요청)와 SSE(서버 발신 이벤트, 스트리밍)로 구분된다. | 클라우드에서 운영되는 서비스를 MCP로 연결할 방법이 없다. | Claude Code 공식 문서 서버처럼 설치 없이 URL만으로 즉시 연결 가능한 서버를 지원한다. | stdio transport(서버를 로컬에 설치), REST API 직접 호출 | 다른 도시 지점에 화상통화하는 것. 물리적으로 같은 공간에 없어도 연결된다. |
| **`mcp__<서버>__<도구>`** | MCP 도구의 이름 규칙. `mcp__github__list_issues`처럼 서버명과 도구명을 `__`로 연결한다. `allowed_tools`에 이 형식으로 허용 목록을 추가한다. | 어떤 도구가 어느 MCP 서버 소속인지 구분할 수 없다. | 여러 MCP 서버의 도구들이 이름 충돌 없이 공존하게 한다. 와일드카드(`mcp__github__*`)로 서버 전체 허용도 가능하다. | 없음 (SDK 명명 규칙) | 직원 사번 체계. `서울-영업팀-001` 형식이면 어느 도시, 어느 팀 직원인지 한눈에 알 수 있다. |
| **`.mcp.json`** | 프로젝트 루트에 두는 MCP 서버 설정 파일. 코드 수정 없이 환경 변수 치환(`${GITHUB_TOKEN}`)을 지원한다. | MCP 서버 설정이 코드에 하드코딩되어 팀 간 공유가 어렵다. | 팀 공통 MCP 설정을 버전 관리하고, 개인 인증 정보는 환경 변수로 분리한다. | `ClaudeAgentOptions(mcp_servers={...})`로 코드에서 직접 설정 | 팀 공용 공구함. 어떤 공구(MCP 서버)가 어디 있는지 팀원 모두가 아는 공유 목록. |

## 8. 스트리밍

텍스트가 완성되기 전에 실시간으로 수신하는 기술.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **Streaming (스트리밍)** | 서버에서 데이터를 한 번에 전부 보내지 않고 생성되는 대로 조금씩 전송하는 방식. Claude는 텍스트를 생성하면서 동시에 전송한다. | 응답이 완성될 때까지 화면이 비어 있다. 긴 응답에서 사용자는 수 초~수십 초를 빈 화면만 본다. | TTFB를 최소화해 첫 글자가 즉시 표시되므로 사용자는 에이전트가 "일하고 있음"을 즉각 인지한다. | 폴링(완료 여부를 주기적으로 확인), 웹소켓(양방향) | 라이브 스포츠 중계. 경기가 끝난 뒤 하이라이트만 보는 것(블로킹)과 실시간으로 보는 것(스트리밍)의 차이. |
| **`include_partial_messages`** | `ClaudeAgentOptions`에 설정하는 불리언 값. `True`로 설정하면 텍스트 생성 중에 `StreamEvent`가 yield된다. 기본값은 `False`. | 스트리밍이 비활성화되어 완성된 메시지만 수신한다. | 토글 하나로 블로킹 방식과 스트리밍 방식을 전환한다. | 없음 (이 옵션이 스트리밍을 활성화하는 유일한 방법) | 수도꼭지. 닫으면(False) 물이 한꺼번에 담아서 주고, 열면(True) 흐르는 대로 바로 흘러나온다. |
| **TTFB (Time To First Byte)** | 첫 번째 데이터(여기서는 첫 글자)가 클라이언트에 도달하기까지 걸리는 시간. 스트리밍의 핵심 성능 지표다. | 측정하지 않으면 스트리밍의 효과를 정량적으로 파악할 수 없다. | UX 품질의 핵심 지표. TTFB가 낮을수록 에이전트가 빠르게 반응하는 것처럼 느껴진다. | 총 응답 시간(Total Response Time) — 완성 기준 | 식당에서 주문 후 음식이 나오기까지의 시간. 전체 요리 시간과 별개로, 첫 번째 반찬이 나오는 시간이 만족도에 크게 영향한다. |
| **`content_block_delta`** | `StreamEvent.event["type"]`의 한 값. 텍스트 청크가 도착할 때 이 타입으로 온다. `event["delta"]["type"] == "text_delta"`이면 `event["delta"]["text"]`에 텍스트가 담겨 있다. | 스트리밍 이벤트에서 실제 텍스트를 어떻게 추출하는지 알 수 없다. | 원시 스트림 이벤트에서 텍스트를 안전하게 추출하는 기준점을 제공한다. | `AssistantMessage`(완성된 텍스트, 지연 있음) | 퍼즐 한 조각. `AssistantMessage`가 완성된 퍼즐이라면, `content_block_delta`는 조각 하나씩 건네받는 것이다. |

## 9. 에이전트 패턴

에이전트가 복잡한 작업을 수행하는 방식과 전략들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **에이전트 루프 (Agent Loop)** | "프롬프트 수신 → 추론 → 도구 호출 → 결과 관찰 → 재추론 → ..."을 최종 응답이 나올 때까지 반복하는 사이클. SDK가 자동으로 실행한다. | 도구 호출 한 번으로는 복잡한 작업을 완결할 수 없다. | 단순 Q&A를 넘어 여러 단계의 행동이 필요한 자율 작업을 가능하게 한다. | 단일 API 호출(`messages.create`) + 수동 루프 구현 | 셰프가 요리를 완성하기까지의 과정. 재료 확인 → 손질 → 조리 → 맛 확인 → 조미 → ... 완성품이 나올 때까지 반복. |
| **ReAct 패턴** | **Re**asoning + **Act**ing의 합성어. "생각(Thought) → 행동(Act) → 관찰(Observe)"을 반복하는 가장 보편적인 에이전트 패턴. Claude Agent SDK의 기본 동작 방식이다. | 목적 없이 무작위로 도구를 호출하거나, 도구 결과를 반영하지 않고 다음 행동을 결정한다. | 추론과 행동을 명시적으로 연결해 에이전트가 맥락에 맞는 다음 행동을 선택하게 한다. | Chain of Thought(행동 없이 추론만), 직접 도구 호출(추론 없이 행동만) | 탐정 수사. "이 단서가 의미하는 건(Thought) → 저 장소를 조사해야겠다(Act) → 단서를 발견했다(Observe) → 따라서 용의자는..."의 반복. |
| **Plan-and-Execute** | **먼저 전체 계획을 수립**한 뒤 각 단계를 순서대로 실행하는 패턴. 즉흥적인 ReAct와 달리 전체 그림이 먼저 확정된다. | 작업이 길어지면 방향을 잃고 이미 한 작업을 반복하거나 중요한 단계를 건너뛰기 쉽다. | 복잡하고 긴 작업에서 전체 범위와 순서를 미리 확정해 일관성 있는 실행을 보장한다. | ReAct(즉흥), 계층적 계획(상위·하위 분리) | 건축 설계 후 시공. 설계도(계획) 없이 짓기 시작하면 나중에 구조를 바꾸기 어렵다. |
| **Chain of Thought (CoT)** | 최종 답변 전에 중간 추론 단계를 명시적으로 거치도록 유도하는 기법. `system_prompt`에 단계별 사고를 지시하거나 Extended Thinking을 활성화한다. | Claude가 어렵고 다단계 추론이 필요한 문제에서 성급하게 답을 낼 수 있다. | 복잡한 수학 문제, 다단계 논리 추론, 코드 디버깅에서 정확도를 높인다. | Extended Thinking(`thinking.budget_tokens`), 자기 비판 루프 | 수학 시험에서 "풀이 과정을 반드시 쓰시오". 정답에 이르는 사고 과정을 강제한다. |
| **계층적 계획 (Hierarchical Planning)** | **Orchestrator(총괄 에이전트)**가 전체 계획을 세우고, **Worker(전문 에이전트)**들이 각 서브태스크를 담당하는 구조. SDK의 `AgentDefinition`으로 구현한다. | 하나의 에이전트가 전문 영역이 다른 여러 작업을 동시에 처리하면 품질이 저하된다. | 역할 분리로 각 에이전트가 전문 영역에 집중하고, 총괄이 조율한다. | 단일 에이전트 + 긴 시스템 프롬프트, 여러 독립 `query()` 호출 | 프로젝트 매니저 + 전문가 팀. PM은 일정을 관리하고, 개발자·디자이너·QA가 각 영역을 담당한다. |
| **Subagent (서브에이전트)** | 메인 에이전트가 특정 작업을 위임하는 **하위 에이전트**. `Agent` 도구로 호출하며, `AgentDefinition`에 이름·지침·사용 도구를 정의한다. | 복잡한 작업 전체를 하나의 에이전트에 맡기면 집중도가 떨어지고 오류가 많아진다. | 전문화된 하위 에이전트에 위임해 더 높은 품질의 결과를 얻는다. | 단일 에이전트 + 역할별 프롬프트, `asyncio.gather`로 병렬 `query()` 실행 | 외주. 법무는 법무법인에, 세무는 세무사에 맡기는 것처럼 전문성에 따라 분리한다. |
| **Map-Reduce** | 여러 입력에 동일한 처리(Map)를 **병렬** 적용한 뒤, 결과를 하나로 합산(Reduce)하는 패턴. "모든 파일 개별 분석 → 종합 보고서"에 적합하다. | 파일 10개를 순차 분석하면 10배의 시간이 걸린다. | 독립적인 처리 작업을 병렬화해 시간을 대폭 줄인다. | 순차 루프, 계층적 서브에이전트 | 선거 개표. 각 지역에서 동시에 표를 세고(Map), 마지막에 전국 결과를 합산(Reduce)한다. |
| **Critic Agent (비평 에이전트)** | 작업 결과를 검토하는 **독립적인 에이전트**. 작업 에이전트와 **다른 세션**에서 실행해 편향 없이 결과를 평가한다. | 자기 검토는 자신의 실수를 찾지 못하는 경향이 있다. | 독립적인 관점에서 오류·누락·개선점을 발견해 결과 품질을 높인다. | 자기 검토 루프(Self-reflection), 테스트 실행(객관적이지만 범위 제한) | 편집자. 작가는 자신의 글에서 오탈자를 잘 못 찾지만 편집자는 쉽게 찾는다. |
| **Iterative Refinement (반복 개선)** | Critic의 피드백을 반영해 결과를 수정하고 다시 검토받는 과정을 **품질 기준을 충족할 때까지** 반복하는 패턴. | 한 번의 실행으로 완벽한 결과를 기대하면 품질이 낮아진다. | 첫 시도의 한계를 인정하고, 피드백 루프로 점진적으로 품질을 끌어올린다. | 단일 실행 + 수동 검토, Extended Thinking으로 첫 시도 품질 향상 | 글쓰기 초고 → 피드백 → 수정의 반복. 한 번에 완벽한 글을 쓰는 작가는 없다. |

## 10. 훅 (Hooks)

에이전트 루프의 특정 시점에 실행되는 커스텀 콜백 함수.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **Hook (훅)** | 에이전트 루프의 특정 이벤트(도구 실행 전후, 세션 시작/종료 등)에 자동으로 호출되는 사용자 정의 함수. `hooks` 옵션에 등록한다. | 에이전트 내부에서 무슨 일이 일어나는지 관찰하거나 개입할 방법이 없다. | 로깅, 감사, 동적 권한 제어, 행동 수정 등을 에이전트 코드 수정 없이 외부에서 주입한다. | `can_use_tool` 콜백(권한 제어 특화), 도구 실행 결과를 `AssistantMessage`에서 추적 | 카메라. 무슨 일이 일어나는지 기록하면서 필요하면 경보를 울릴 수도 있다. |
| **`PreToolUse`** | **도구 실행 직전**에 호출되는 훅. 도구명과 입력값을 받아 실행을 허용·수정·차단할 수 있다. 빈 dict 반환 = 허용, 특수 응답 = 차단. | 위험한 도구 실행을 사전에 막을 수 없다. | 실행 전 검증, 입력값 sanitization, 동적 차단 로직을 구현한다. | `can_use_tool` 콜백, `disallowed_tools` (정적 차단) | 출입 전 보안 검사. 가방을 X선으로 검사하고 위험물이 있으면 통과를 막는다. |
| **`PostToolUse`** | **도구 실행 직후**에 호출되는 훅. 도구명, 입력값, 도구 응답을 받아 로깅·알림 등 사후 처리를 할 수 있다. | 어떤 도구가 실행됐고 결과가 무엇인지 외부에서 추적할 수 없다. | 감사 로그, 실행 통계, 결과 기반 알림 등 사후 처리 로직을 구현한다. | `AssistantMessage`의 `ToolUseBlock` 추적(도구 호출 기록은 되지만 결과는 미포함) | 결제 후 영수증. 거래가 완료된 뒤 내역을 기록하는 것. |
| **`HookMatcher`** | 훅을 어떤 도구에 적용할지 지정하는 설정 객체. `matcher` 필드에 정규식 패턴을 쓰면 해당 도구에만 훅이 적용된다. `matcher=None`이면 모든 도구에 적용. | 모든 도구에 훅이 적용되어 불필요한 로그가 쌓이거나 성능이 저하된다. | "Edit과 Write 도구에만 감사 로그를 남겨라"처럼 훅 적용 범위를 세밀하게 제어한다. | 훅 콜백 내부에서 도구명으로 직접 분기 처리 | 특정 부서에만 적용되는 사내 규정. "임원급 결재"가 필요한 항목을 선별하는 것. |
| **`HookCallback`** | 훅이 실제로 실행하는 비동기 함수의 타입. `(input_data, tool_use_id, context) → dict`를 반환한다. 빈 dict `{}`를 반환하면 정상 진행. | 없음 (HookCallback은 훅 자체다) | — | — | 도어맨. 손님(도구 실행)이 문을 통과하게 할지, 확인 절차를 밟게 할지, 돌려보낼지 결정하는 사람. |

## 11. 출력 형식 & 프롬프트

Claude의 응답 형식과 행동 방식을 제어하는 도구들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **`system_prompt`** | 대화 전체에 적용되는 Claude의 역할과 행동 지침. 모든 user 메시지보다 우선하며 대화 내내 유지된다. | Claude가 범용적으로 응답하며, 도메인 특화 행동을 기대하기 어렵다. | "너는 Python 전문가다", "항상 한국어로 답해라" 같은 고정 행동 규칙을 전역으로 설정한다. | 각 `query()` 프롬프트에 역할 지시 포함 (매번 반복해야 함) | 직원 채용 공고의 "직무 기술서". 이 직원은 계약 기간 내내 이 역할을 수행한다. |
| **`output_format` / Structured Output** | JSON 스키마를 지정해 Claude의 응답이 정해진 형식을 따르도록 강제하는 기능. `ResultMessage.structured_output`에 파싱된 dict가 담긴다. | Claude가 자유 형식 텍스트로 응답해 프로그래밍적 처리를 위해 파싱이 필요하다. | 응답을 바로 `dict`로 사용할 수 있어 추가 파싱 없이 워크플로우 다음 단계로 넘길 수 있다. | 프롬프트에 "JSON으로 답해주세요" 지시(스키마 강제 미보장), JSON 파싱 시도 | 공식 서류 양식. 자유 서술 대신 정해진 칸에 정해진 형식으로 작성하도록 강제한다. |
| **JSON Schema** | `output_format`에 전달하는 응답 구조 정의. 필드명, 타입, 필수 여부, 열거 값 등을 명세한다. | 스키마 없이는 응답 형식을 보장할 수 없다. | 응답 구조를 컴파일 타임 수준으로 엄격하게 정의해 런타임 오류를 예방한다. | Pydantic 모델(Python), Zod(TypeScript) — SDK 외부에서 검증 | 데이터베이스 테이블 스키마. 어떤 컬럼이 어떤 타입으로 들어와야 하는지 미리 정의한다. |
| **`SystemPromptPreset`** | Claude Code의 기본 시스템 프롬프트를 재사용하는 옵션. `{"type": "preset", "preset": "claude_code"}`로 지정하며, `append`로 추가 지침을 덧붙일 수 있다. | Claude Code 기본 설정을 에이전트에 적용하려면 전체 프롬프트를 복사해야 한다. | Claude Code의 풍부한 기본 지침을 그대로 유지하면서 추가 지침만 덧붙일 수 있다. | 커스텀 `system_prompt` 직접 작성 | 회사 업무 매뉴얼 + 팀별 추가 규정. 공통 규정을 복사하지 않고 참조한 뒤 팀 규정만 추가한다. |
| **Extended Thinking** | `thinking` 옵션 활성화 시 Claude가 **더 긴 내부 추론**을 수행하는 기능. `budget_tokens`로 사고에 쓸 토큰 한도를 지정한다. `ThinkingBlock`으로 사고 과정이 노출된다. | 빠른 응답만 필요할 때는 오히려 불필요하게 느리고 비용이 높다. | 수학·논리·복잡한 코드 분석처럼 깊은 추론이 필요한 작업에서 정확도를 크게 높인다. | Chain of Thought 프롬프트(추론 과정을 명시적으로 지시), `effort` 옵션 | 시험 전 충분한 준비 시간. 바로 답을 쓰는 것과 30분 더 생각하고 쓰는 것의 차이. |

## 12. 비용 & 제한

에이전트 실행의 자원 소비를 측정하고 제어하는 개념들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **Token (토큰)** | Claude가 텍스트를 처리하는 최소 단위. 영어 단어 하나가 대략 1~1.3 토큰, 한국어는 글자당 약 2~3 토큰이다. API 비용과 컨텍스트 창 크기의 기준이다. | 없음 (기본 단위이므로 대체 불가) | 텍스트를 고정 크기 단위로 나눠 처리·과금·제한을 일관되게 적용할 수 있게 한다. | 없음 | 화폐의 기본 단위(원, 센트). 모든 거래(API 호출)의 가격이 이 단위로 측정된다. |
| **`total_cost_usd`** | `ResultMessage`에 담기는 해당 에이전트 루프의 **총 비용** (미국 달러). 입력·출력 토큰, 도구 실행, 웹 검색 등 모든 비용의 합산이다. | 에이전트가 얼마의 비용을 소모했는지 추적할 수 없다. | 단계별·에이전트별 비용을 추적해 비용 최적화와 예산 초과 감지를 가능하게 한다. | `usage` 필드의 토큰 수 × 단가 계산 | 영수증 합계 금액. 각 항목을 일일이 더하지 않아도 총액을 바로 알 수 있다. |
| **`max_turns`** | 에이전트 루프의 최대 도구 호출 횟수. 이 값을 초과하면 `ResultMessage.subtype == "error_max_turns"`로 종료된다. | 에이전트가 무한 루프에 빠지거나 필요 이상으로 많은 도구를 호출해 비용이 폭등한다. | 예상 밖의 장시간 실행을 차단하고 비용을 제어한다. | `max_budget_usd`(비용 기준 제한), 타임아웃(시간 기준) | 요리사에게 "재료는 최대 10가지만 써라"는 제한. 무한정 재료를 쓰지 않도록 범위를 확정한다. |
| **`max_budget_usd`** | 에이전트 루프에서 소비 가능한 최대 비용(달러). 이 값을 초과하면 `error_max_budget_usd`로 종료된다. | 고비용 에이전트가 계획보다 훨씬 많은 비용을 소모할 수 있다. | 예산 초과를 자동으로 방지한다. 단계별로 다른 예산을 설정해 비용을 세밀하게 관리할 수 있다. | `max_turns`(횟수 기준), 외부 모니터링 + 강제 종료 | 법인 카드 한도. 한도에 도달하면 자동으로 결제가 차단된다. |
| **`model`** | 사용할 Claude 모델 ID. `"claude-opus-4-7"`, `"claude-sonnet-4-6"`, `"claude-haiku-4-5"` 등. 성능과 비용이 비례한다. | SDK 기본 모델이 사용되어 원하는 성능/비용 트레이드오프를 적용할 수 없다. | 작업 복잡도에 맞는 모델을 선택해 비용을 최적화한다. 간단한 작업에 비싼 모델을 쓸 필요가 없다. | `effort` 옵션(모델 대신 노력 수준으로 제어) | 택시 vs 지하철 vs 도보. 목적지와 중요도에 따라 교통수단을 선택하는 것. |
| **`effort`** | `"low"`, `"medium"`, `"high"`, `"xhigh"`, `"max"` 5단계로 에이전트의 작업 깊이를 제어하는 옵션. 내부적으로 thinking 예산, 모델 등을 자동 조정한다. | 모든 작업에 최대 성능 모델을 쓰면 비용이 과다 발생한다. | 작업 중요도에 따라 "빠르고 저렴하게" 또는 "깊고 정확하게"를 한 번에 설정한다. | `model` + `thinking` 직접 조합 | 자동차 드라이브 모드. 에코 모드(low)부터 스포츠 모드(max)까지 한 번의 설정으로 전환한다. |

## 13. 워크플로우 & 고급 개념

자율 에이전트 시스템을 설계할 때 필요한 아키텍처와 전략 개념들.

| 용어 | 개념 설명 | 없으면 발생하는 문제 | 해결하는 문제 | 대체 수단 | 일상 비유 |
|------|-----------|----------------------|---------------|-----------|-----------|
| **RAG (Retrieval-Augmented Generation)** | 생성 전에 외부 지식 저장소에서 관련 문서를 **검색(Retrieval)**해 프롬프트에 주입하는 패턴. 벡터 DB(Chroma, Pinecone 등)를 주로 사용한다. | Claude의 학습 데이터에 없는 도메인 지식을 활용할 수 없다. | 최신 문서, 내부 문서, 도메인 특화 지식을 Claude에 실시간으로 주입해 정확한 답변을 유도한다. | 파일 전체를 프롬프트에 포함(컨텍스트 창 한계), WebFetch(URL이 알려진 경우) | 시험 전 참고서 찾기. 모든 걸 외우는 대신 필요할 때 관련 챕터를 펼쳐 참고한다. |
| **벡터 DB (Vector Database)** | 텍스트를 수치 벡터(임베딩)로 변환해 저장하고, **의미적 유사성**으로 검색하는 데이터베이스. Chroma, Pinecone, pgvector, Qdrant 등이 대표적이다. | 키워드 검색만 가능해 "인증 관련 코드"처럼 의미 기반 검색이 안 된다. | "이 문서와 비슷한 내용" 같은 시맨틱 검색으로 더 관련성 높은 문서를 찾는다. | 키워드 검색(Grep, ElasticSearch), 전체 텍스트 직접 포함 | 도서관 사서의 추천. "이 책을 읽은 사람은 이 책도 봤어요." 주제·내용의 유사성으로 추천한다. |
| **Structured Output** | `output_format`으로 JSON 스키마를 지정해 Claude 응답이 항상 파싱 가능한 구조를 갖게 하는 기능. `ResultMessage.structured_output`에서 `dict`로 바로 접근한다. | 자유 텍스트 응답을 파싱하다 실패하거나 포맷이 매번 달라진다. | 에이전트 출력을 다음 처리 단계의 안정적인 입력으로 만든다. 에이전트 간 데이터 교환에 필수. | 프롬프트로 JSON 유도 + 수동 파싱, Pydantic/Zod 검증 추가 | 세금 신고 표준 양식. 사람마다 다른 형식으로 제출하면 처리할 수 없다. |
| **Human-in-the-Loop** | 에이전트 실행 중 특정 시점에 사람이 개입해 결정하거나 승인하는 설계 패턴. `AskUserQuestion` 도구나 `permission_mode="default"`가 이를 구현한다. | 에이전트가 중요한 결정을 임의로 내린다. 돌이킬 수 없는 행동(파일 삭제, 이메일 전송)이 실수로 실행될 수 있다. | 고위험 행동 전에 사람의 최종 확인을 받아 자동화와 안전성을 동시에 달성한다. | 사전 승인된 허용 목록만 실행(`allowed_tools`), 사후 검토(롤백 가능한 경우) | 의약품 처방. 약사가 자동으로 조제하기 전에 의사의 처방전(사람의 승인)이 필요하다. |
| **`setting_sources`** | SDK가 설정 파일을 로드할 소스를 제한하는 옵션. `"user"` (`~/.claude/settings.json`), `"project"` (`.claude/settings.json`), `"local"` (`.claude/settings.local.json`)을 선택적으로 포함한다. | 의도하지 않은 로컬 설정이 에이전트 동작에 영향을 준다. CI 환경에서 예측 불가능한 동작 발생. | 에이전트가 로드할 설정 소스를 명시적으로 선언해 환경 간 동작 일관성을 보장한다. | 없음 (기본값은 모든 소스 로드) | 실행 환경 격리. 개발자 로컬 설정이 운영 서버에 영향을 주지 않도록 환경을 분리하는 것. |
| **Audit Trail (감사 추적)** | 에이전트가 어떤 도구를 언제, 어떤 인자로 호출했는지 기록하는 로그. `PostToolUse` 훅과 세션 파일(`.jsonl`)이 이를 제공한다. | 에이전트가 왜 그런 결과를 냈는지 소급 확인이 불가능하다. 오류 재현·디버깅이 어렵다. | 규정 준수(Compliance), 디버깅, 에이전트 행동 분석을 위한 완전한 실행 기록을 유지한다. | 세션 파일 직접 분석(`get_session_messages()`), 외부 APM 도구 | CCTV 녹화. 무슨 일이 있었는지 나중에 돌려볼 수 있다. |
| **`cwd` (Current Working Directory)** | 에이전트가 파일 시스템 작업을 수행하는 기준 디렉토리. 상대 경로의 해석 기준이 된다. 세션 파일도 이 경로 기준으로 저장된다. | 에이전트가 예상 외의 디렉토리에서 파일을 읽거나 수정한다. 세션 resume 시 잘못된 위치에서 파일을 찾는다. | 에이전트의 작업 범위를 특정 디렉토리로 제한하고 세션 위치를 결정한다. | 없음 (기본값은 프로세스 실행 디렉토리) | 사무실 데스크. 직원이 일하는 기준 위치. "이 파일 가져다줘"라고 하면 이 책상 기준으로 찾는다. |
| **Pipeline (파이프라인)** | 여러 에이전트 실행 단계를 **선형으로 연결**해 이전 단계 출력이 다음 단계 입력이 되는 구조. 각 단계는 독립적이고 책임이 명확하다. | 모든 작업을 하나의 거대한 에이전트에 맡기면 중간 단계 실패 시 전체를 재실행해야 한다. | 단계별 독립 실행, 부분 재실행, 결과 캐싱이 가능해 복잡한 작업의 신뢰성을 높인다. | 단일 `query()` + 긴 프롬프트, 서브에이전트 계층 구조 | 공장 컨베이어 벨트. 각 스테이션이 특정 공정만 담당하고, 결과물이 다음 스테이션으로 넘어간다. |
